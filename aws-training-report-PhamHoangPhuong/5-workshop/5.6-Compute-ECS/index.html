5.6-Compute-ECS

Step 8 — Amazon ECR + Push Container Images

Push your container images to Amazon ECR, configure an Application Load Balancer at the front, and run the 3 decoupled services on Amazon ECS Fargate backed by target-tracking Auto Scaling.

Navigate to Amazon ECR → select Private registry -> Repositories -> click Create repository: Create 3 individual repositories: saashr-auth, saashr-tenant, and saashr-hr.

  • The 3 container repositories are successfully provisioned and ready.

  • Build and push each microservice image sequentially (using the auth service as an example):
  • # Authenticate your local Docker CLI against the private AWS ECR registry
aws ecr get-login-password --region ap-southeast-1 | docker login --username AWS --password-stdin <acct>.dkr.ecr.ap-southeast-1.amazonaws.com

# Build the local Docker image
docker build -t saashr-auth ./microservices/auth-service

# Tag the image referencing your remote ECR repository path
docker tag saashr-auth:latest <acct>.dkr.ecr.ap-southeast-1.amazonaws.com/saashr-auth:latest

# Push the final container image to Amazon ECR
docker push <acct>.dkr.ecr.ap-southeast-1.amazonaws.com/saashr-auth:latest

  • Executing the push command via the terminal (example showing the hr service upload):

  • The 3 ECR repositories now display their freshly pushed container images tagged as latest:

Step 9 — Application Load Balancer Configuration

Configure individual Target Groups (Target type must be set to IP for compatibility with Amazon ECS Fargate tasks, mapping one target group to each microservice):

  • Target Groups to provision: tg-auth (Port: 8000), tg-tenant (Port: 8001), and tg-hr (Port: 8002).
  • Health check configurations: Set paths to /api/v1/auth/health, /api/v1/tenants/health, and /api/v1/hr/health respectively.
  • Optimization: Set the Healthy threshold count to 2 (Reduces evaluation cycles to roughly 60 seconds -> facilitates faster service deployments).

  • Proceed with the creation of tg-tenant and tg-hr target groups following the same blueprint:

  • Navigate to EC2 → Load Balancers → click Create Load Balancer → Select Application Load Balancer -> click Create.

  • Configure basic load balancer settings:
    • Name: saashr-alb
    • Scheme: Internet-facing
    • IP address type: IPv4

  • Network mapping:
    • VPC: Select saashr-vpc
    • Subnets mapping: Explicitly check public-1a + public-1b
    • Security Groups: Assign sg-alb

  • Listeners and Routing:
    • Protocol: HTTP
    • Port: 80
    • Default Routing Action: Forward to target group → tg-auth -> click Create load balancer.

  • The Application Load Balancer is successfully created.
  • Configure Path-Based Routing Rules: Modify the HTTP:80 Listener rules to implement the following conditional paths:
    • Requests to /api/v1/auth/* → Forward to tg-auth
    • Requests to /api/v1/tenants/* → Forward to tg-tenant
    • Requests to /api/v1/hr/* → Forward to tg-hr

  • Click into the HTTP:80 Listener configuration to manage routing rules:

  • Click Add Rule:
  • Add Condition -> Path: /api/v1/auth/*

  • Select Action -> Forward to target group: tg-auth

  • Repeat this sequence to establish routing paths for the remaining microservices. Final routing rules structure appears as follows:

Step 10 — Amazon ECS Fargate + Auto Scaling Configuration

Navigate to Amazon ECS → click Create cluster → Name: saashr-cluster, Infrastructure: AWS Fargate (Serverless compute).

  • Task Definitions Provisioning: Create an individual Task Definition for each microservice using Fargate launch type, allocated with 0.25 vCPU and 0.5 GB Memory.
    • Container Image: Paste the corresponding image URI from your ECR repository.
    • Port Mappings: Match the precise operational port of the backend application.
    • Task Role (Least Privilege Application Permissions): Grants permissions such as sqs:SendMessage/ReceiveMessage/DeleteMessage for the tenant and HR workers, cognito-idp:* minimum permissions for the auth component, and ssm:GetParameters.
    • Task Execution Role: Permits the ECS agent to pull container images from ECR, stream runtime outputs to CloudWatch Logs, and securely decrypt credentials from the SSM Parameter Store.
    • Configuration Injection: Secrets are referenced dynamically from the SSM Parameter Store; the log driver is set to awslogs routing output to log groups named /ecs/saashr/{service}.
  • 1. Create Task Definition for Auth Service:
    • Family Name: saashr-auth
    • CPU: .25 vCPU
    • Memory: .5 GB

    • Task Role: None (Auth does not interact directly with AWS resources outside of Cognito)
    • Task Execution Role: ecsTaskExecutionRole
    • Container Name: saashr-auth
    • Image URI: Copy the URI string from the saashr-auth ECR repository.
    • Container Port: 8000

  • Environment Variables Configuration -> Click Add environment variable and bind container keys to their respective Value From targets pointing to SSM Parameter Store keys as shown below:

  • 2. Create Task Definition for Tenant Service:
    • Family Name: saashr-tenant
    • CPU: .25 vCPU
    • Memory: .5 GB

    • Task Role: Click to create a new IAM role:
        - Role Name: saashr-tenant-task-role
        - Attached Custom Policy: Replace template s3:GetObject actions with sqs:SendMessage and replace resource path strings with your primary SQS ARN: arn:aws:sqs:ap-southeast-1:016461465939:saashr-events
        - Click Create role.

    • Task Role Binding: Assign saashr-tenant-task-role
    • Task Execution Role: ecsTaskExecutionRole
    • Container Name: saashr-tenant
    • Image URI: Copy the URI string from the saashr-tenant ECR repository.
    • Container Port: 8001

  • Environment Variables Configuration -> Map internal keys to the corresponding SSM Parameter Store locations as shown:

  • 3. Create Task Definition for HR Service:
    • Family Name: saashr-hr
    • CPU: .25 vCPU
    • Memory: .5 GB

    • Task Role: Click to create a new IAM role:
        - Role Name: saashr-hr-task-role
        - Attached Custom Policy: Replace S3 permissions with sqs:ReceiveMessage, sqs:DeleteMessage, and sqs:GetQueueAttributes, targeting the SQS queue ARN: arn:aws:sqs:ap-southeast-1:016461465939:saashr-events
        - Click Create role.

    • Task Role Binding: Assign saashr-hr-task-role
    • Task Execution Role: ecsTaskExecutionRole
    • Container Name: saashr-hr
    • Image URI: Copy the URI string from the saashr-hr ECR repository.
    • Container Port: 8002

  • Environment Variables Configuration -> Bind runtime keys to SSM Parameter Store configurations as shown:

  • Service Auto Scaling Configuration (Applied to each running service): Implement a Target Tracking scaling policy tied to the metric ECSServiceAverageCPUUtilization set to a target value of 70%. Capacity parameters are defined with a Minimum task count of 2 and a Maximum task count of 4, enabling both automatic scale-out and scale-in actions.
