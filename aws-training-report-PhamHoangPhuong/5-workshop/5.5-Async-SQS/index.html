5.5-Async-SQS

Step 6 — Amazon SQS + DLQ (Asynchronous Queueing)

The tenant-service publishes core business events, while the hr-service consumes them asynchronously. This queueing mechanism decouples both microservices, absorbs high traffic spikes, and leverages a Dead-Letter Queue (DLQ) to isolate failing or corrupted messages.

  • Search for and open Simple Queue Service → select Queues from the left sidebar.

  • Order of Operations Note: The AWS Console requires a Dead-Letter Queue to exist before it can be assigned to a primary queue. Therefore, we must create saashr-events-dlq first, and then configure the main saashr-events queue to route undeliverable messages to it.
  • 1. Create the Dead-Letter Queue (saashr-events-dlq):
    • Type: Standard
    • Name: saashr-events-dlq
    • Access policy: Advanced
    • Click Create queue.

  • 2. Create the Primary Queue (saashr-events):
    • Type: Standard
    • Name: saashr-events
    • Keep default configurations, except for setting Receive message wait time = 20 seconds (Enables SQS Long-Polling to optimize consumer efficiency and reduce API costs).
    • Access policy: Advanced
    • Dead-letter queue panel → Select "Set this queue to receive undeliverable messages" = Enabled
    • Choose queue: Choose the ARN corresponding to arn:aws:sqs:ap-southeast-1:<account-id>:saashr-events-dlq
    • Maximum receives: 5 (The message will automatically drop into the DLQ after 5 consecutive failed processing retries).
    • Click Create queue.

  • Copy the unique Queue URL generated for saashr-events → save it into the SSM Parameter Store under the path /saashr/sqs/url (as detailed in Step 4).
  • Application Source Code Implementations (Pre-configured):
  • Producer Side: Embedded inside tenant-service/app/core/sqs.py (Leverages the boto3 SDK to securely inherit IAM Execution credentials from the active ECS task-role, removing the need for static access keys):
    • Code Logic: send_message(QueueUrl=SQS_QUEUE_URL, MessageBody=json.dumps(event))
  • Consumer Side: Driven by hr-service/app/core/worker.py via the start_sqs_consumer loop: Polls SQS via long-polling → executes business computations → explicitly triggers delete_message upon successful execution. In case of runtime failure, the message undergoes automatic retry routines before gracefully dropping into the DLQ.
  • 📎 Attachments: sqs.py, worker.py (Place inside 5.5-Async-SQS/files/).
  • Implementing this asynchronous, event-driven pattern via Amazon SQS acts as a structural buffer, offering massive horizontal elasticity across decoupled application layers.
