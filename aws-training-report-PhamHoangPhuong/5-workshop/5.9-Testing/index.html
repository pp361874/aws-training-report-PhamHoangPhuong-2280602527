1. Health Check (Smoke Testing)

Executing end-to-end stack validation across the deployed architecture via the centralized CloudFront ingress gateway, followed by failure testing, high-availability evaluation, and observability tracking. Define the base environment URL:

BASE="https://d3htvmot332c6v.cloudfront.net/"

curl -s $BASE/api/v1/auth/health
curl -s $BASE/api/v1/tenants/health
curl -s $BASE/api/v1/hr/health

Verification Results: All three microservices return a healthy response (the "database": "up" state explicitly confirms that both the horizontal ECS application tasks and the underlying RDS storage clusters are accessible):

{"status":"healthy","service":"auth-service"}
{"status":"healthy","database":"up","service":"tenant-service"}
{"status":"healthy","database":"up","service":"hr-service"}

2. Frontend Delivery Validation

curl -I $BASE/

Verification Results: Returns an HTTP/2 200 OK response with content-type: text/html—confirming the React SPA application is properly bundled, managed, and delivered via CloudFront edge locations from the secure, private S3 storage origin.

3. End-to-End Authentication Workflow

# 1. Provision a new tenant workspace and master owner identity:

curl -s -X POST $BASE/api/v1/auth/register -H "Content-Type: application/json" \
  -d '{"email":"owner@acmedemo.com","password":"Passw0rd!","tenant_name":"Acme Demo","subdomain":"acmedemo"}'

# 2. Authenticate the tenant session to fetch an secure access token:

curl -s -X POST $BASE/api/v1/auth/login -H "Content-Type: application/json" \
  -d '{"email":"owner@acmedemo.com","password":"Passw0rd!","subdomain":"acmedemo"}'

# 3. Request identity context parsing utilizing the returned token:

curl -s $BASE/api/v1/auth/me -H "Authorization: Bearer $TOKEN"

Verification Results: The /me routing path successfully reconstructs the authenticated user payload tied to the respective permission role maps:

{"id":"usr_9476d71e-...","email":"owner@acmedemo.com","status":"active","active_tenant_id":"tenant_369f39d7-...","role":"owner"}

📎 Architectural Context: Tokens are encoded as self-contained RS256 JWTs generated by the application layer. The auth-service signs the outbound payload with its private key, while downstream microservices independently decrypt and evaluate the digital signature using the shared public key repository (as detailed in the Data & Identity section). Inside these JWT claims, corporate tenant_id boundaries and role definitions are enforced across every sequential API call.

5. Exception Handling & Edge-Case Verification

API error boundary states are tested directly against the Application Load Balancer endpoint (http://<alb-dns>/...). This is necessary because CloudFront's custom SPA error handling intercepts and translates 403: Forbidden and 404: Not Found states into /index.html (200) to support client-side routing—which masks native API fault response payloads from testing scripts. Evaluating the API directly at the ALB tier ensures clean monitoring inputs where the response headers explicitly reveal server: uvicorn.

  • Observed Behavior — Duplicate Subdomain Prevention: 
    {"detail":"Tenant subdomain already registered."}

    📎 Identified Edge-Case: Attempting to insert duplicate department names within the same tenant context is successfully blocked at the storage layer via a strict database schema constraint (uq_tenant_dept). However, the application layer currently surfaces a generic HTTP 500 Internal Server Error error payload to the client instead of a cleaner HTTP 409 Conflict. Data integrity and multi-tenant isolation parameters remain completely secure, though API error translation mappings can be improved in subsequent release patches.

6. Asynchronous Messaging Integration Audit (SQS)

End-to-end integration testing of the asynchronous event loop (triggered by modifying tenant lifecycle states → processing events via Amazon SQS → executing background consumption routines in the hr-service) revealed an integration mismatch in the current stack iteration—highlighting the value of this validation checkpoint:

  • Infrastructure Integrity: Injecting mock telemetry messages directly into the saashr-events queue via the AWS Console executes cleanly. Tracking dashboards reflect immediate spikes across NumberOfMessagesSent and SentMessageSize metrics, confirming that primary queues and Dead-Letter Queues (DLQ) are configured properly.
  • Producer Routing: The tenant-service container failed to dispatch transactional event messages to the queue during application-driven state transitions (no MessagesSent data points originated from the software container), despite possessing valid sqs:SendMessage execution permissions within its assigned IAM task role.
  • Consumer Polling: Test payloads staged manually inside the queue were not pulled by the worker framework (metrics displayed MessagesVisible = 1 and MessagesDeleted = 0), indicating that the asynchronous execution worker was not actively processing messages during this test run.
  • Assessment: The structural decoupling architecture blueprint (combining SQS Queue buffering, failure-safe DLQs, and long-polling worker tasks) is validated and sound. The breakdown is located strictly within application-level code hookups (likely caused by stale container image versions or uninitialized background tasks) and is undergoing active remediation. This is documented as an authentic engineering log entry rather than claiming false success.

7. System Observability and Incident Alerting

Log Auditing: Amazon CloudWatch Log Streams mapped to /ecs/saashr-auth, /ecs/saashr-tenant, and /ecs/saashr-hr log groups systematically capture real-time HTTP routing metrics, connection tracking, and unhandled runtime exceptions.

Performance Telemetry: CloudWatch metric charts consolidate real-time CPUUtilization tracking parameters across each active cluster container service.

Incident Routing Alarm Test: To validate notification systems without spinning up artificial compute pressure overhead, simulate an active alarm condition via the AWS CLI:

aws cloudwatch set-alarm-state --alarm-name saashr-alerts-alarm-noti \
  --state-value ALARM --state-reason "manual verification test"

Expected Operational Output: The metrics engine switches to an ALARM state, triggering an immediate Amazon SNS routing routine that dispatches alert summaries to all registered engineer email endpoints within seconds.
