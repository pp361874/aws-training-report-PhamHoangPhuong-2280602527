Workshop

Deploying a Modern 3-Tier SaaS HR Multi-Tenant Microservices Architecture on AWS

Overview

In this workshop, we will build and deploy a SaaS Human Resource Management (HRM) system following a Multi-Tenant Microservices model on the AWS platform. The objective of this workshop is to help learners understand how to design and operate a modern enterprise application capable of serving multiple clients (tenants) on a single platform while ensuring data isolation, scalability, and high availability.

The system is deployed using a 3-Tier Architecture, where the presentation layer is completely decoupled from the business logic and database layers. All components are deployed using AWS Managed Services to minimize infrastructure management overhead, enable automatic scaling, and enhance system reliability.

Throughout this workshop, we will fully deploy all components, including the Frontend, API Gateway, Backend Microservices, Databases, and supporting services such as Identity, Monitoring, Logging, Messaging, and FinOps. Additionally, the workshop introduces modern architectural models widely adopted in the industry, such as Multi-Tenant SaaS, Event-Driven Architecture, Zero Trust Security, High Availability, and Cloud Cost Optimization.

Application Architecture

The application is built on a Microservices model, where each business domain is separated into an independent service to improve scalability, deployment efficiency, and maintainability. The Frontend is developed using ReactJS combined with Vite, providing a Single Page Application (SPA) interface for users to perform functions like logging in, managing employees, departments, timekeeping, leave requests, and monitoring system status. Incoming API requests are handled via an Nginx API Gateway, acting as a Reverse Proxy. It routes requests to the correct backend service, configures CORS, handles SSL Termination, and automatically injects an X-Correlation-ID to trace the entire lifecycle of requests across microservices.

The backend is developed using Python FastAPI and is divided into three main microservices:

  • auth-service: Responsible for user account management, registration, login, security, and authentication. This service integrates with Amazon Cognito to issue JWT Tokens using the RS256 algorithm, containing metadata such as tenant_id and user roles.
  • tenant-service: Manages the enterprises using the system, subscription packages, dedicated subdomains for each tenant, and triggers business events when data changes.
  • hr-service: Handles core HR operations including employee management, departments, timekeeping, work history, and leave requests. This service implements a logical data separation mechanism (Pool Model), ensuring all data queries are strictly bounded by tenant_id.
  • Each microservice owns its own logical database, ensuring independence between services and adhering to the Database per Service principle in Microservices architecture.

Deployment Architecture on AWS

The entire system is deployed in the ap-southeast-1 (Singapore) region using a Web Application 3-Tier model distributed across two Availability Zones (AZs) to enhance fault tolerance. Users access the application through Amazon CloudFront, which serves as the Single Entry Point for the system. CloudFront distributes static ReactJS content from Amazon S3 and forwards API requests to the Application Load Balancer. The production-built frontend is stored securely in an Amazon S3 bucket configured as private, allowing access exclusively from CloudFront via Origin Access Control (OAC) to reinforce security.

API requests forwarded by CloudFront hit the Application Load Balancer, which routes traffic to the corresponding ECS Fargate Services based on the API path (e.g., /api/v1/auth, /api/v1/tenants, and /api/v1/hr). The three microservices are deployed on Amazon ECS Fargate as serverless containers within Private Application Subnets. This removes the operational burden of managing EC2 instances while ensuring independent scaling and deployment. All container images are stored in Amazon Elastic Container Registry (ECR) and are automatically pulled by ECS during new deployments.

Identity Management and Security

The workshop utilizes Amazon Cognito User Pools to manage user identities. Upon successful login, Cognito authenticates the credentials and issues a JWT Token containing claims such as:

  • tenant_id, role, user information

Microservices verify the JWT signature using Cognito's JWKS (JSON Web Key Sets) instead of sharing private keys, enhancing security and simplifying cross-service authentication. Sensitive information such as database passwords, Cognito Client IDs, SQS URLs, and other configuration variables are securely stored in the AWS Systems Manager Parameter Store as SecureString types to avoid hardcoding secrets in the source code.

Database Architecture

System data is securely stored in Amazon RDS MySQL Multi-AZ. To optimize costs while adhering to the Database per Service principle, the system utilizes a single RDS instance but provisions three distinct logical databases:

  • auth_db, tenant_db, hr_db

The Amazon RDS instance is configured in Multi-AZ mode, where data is synchronously replicated from the Primary instance to a Standby instance. If a failure occurs in the primary Availability Zone, the system automatically fails over to the Standby instance without manual intervention. Additionally, automated backups are retained for 7 days to guarantee robust disaster recovery capabilities.

Network Architecture

The infrastructure is provisioned inside an Amazon VPC with a 10.0.0.0/16 CIDR block spanning two Availability Zones. The VPC is segregated into six subnets:

  • Two Public Subnets dedicated for the Application Load Balancer and the NAT Gateway, Two Private Application Subnets dedicated for ECS Fargate tasks, Two Private Database Subnets isolated exclusively for Amazon RDS.

Public Subnets connect directly to the internet via an Internet Gateway. Private Application Subnets leverage a NAT Gateway to securely access external AWS services like Amazon Cognito, Amazon SQS, Amazon ECR, and Amazon CloudWatch without assigning public IPs to the ECS Tasks. Private Database Subnets are completely isolated from the internet to safeguard backend databases from unauthorized external threats.

Asynchronous Communication

This workshop adopts an Event-Driven Architecture to loosen the coupling between microservices. When the tenant-service executes transactions such as creating a new tenant or updating enterprise profiles, it publishes business events to Amazon SQS instead of calling the hr-service directly via synchronous HTTP requests. The hr-service continuously polls the queue to consume and process these messages. Utilizing Amazon SQS allows the system to absorb high traffic spikes, handle background processing efficiently, and increase overall resilience if a target service undergoes temporary downtime. Messages that repeatedly fail processing are automatically re-routed to a Dead Letter Queue (DLQ) for further inspection and troubleshooting.

System Monitoring

All ECS Services stream their application logs and performance metrics to Amazon CloudWatch. CloudWatch tracks critical system states including:

  • CPU Utilization
  • Memory Utilization
  • Container Logs
  • Application Logs
  • ECS Health Status

A CloudWatch Alarm is configured to trigger if CPU utilization exceeds 80%. Once an alarm condition is met, Amazon SNS fires an immediate email notification to the engineering team for rapid incident response.

FinOps and Cost Optimization

The workshop is designed with a FinOps mindset, striking a practical balance between production-grade architecture and operational costs. Rather than deploying two NAT Gateways across both Availability Zones, this workshop utilizes a single shared NAT Gateway for both Private Application Subnets. This significantly cuts down monthly baseline costs and is highly suitable for dev/test and educational environments. The system also utilizes a single Amazon RDS instance to house three logical databases, which maintains the microservice topology while lowering database compute costs.

Deploying backend services on Amazon ECS Fargate removes EC2 management overhead, minimizes maintenance labor, and ensures you only pay for the exact CPU and memory resources consumed by running tasks. For demo environments, the entire infrastructure (NAT Gateway, ALB, RDS) can be completely destroyed post-workshop to bring ongoing costs down to $0. Additionally, AWS Budgets is configured with a threshold of approximately $130/month to track expenditures and alert you before cost overruns happen.

End-to-End System Workflow

Users access the website via Amazon CloudFront. CloudFront delivers static web assets from Amazon S3 and proxies incoming API requests to the Application Load Balancer. The Application Load Balancer routes the requests to the appropriate ECS Fargate Service running in one of the two Availability Zones. The Auth Service authenticates credentials against Amazon Cognito and returns a signed JWT Token containing the user's tenant_id and role permissions. The microservices inspect the JWT to authorize the user and perform data mutations on the Amazon RDS Primary instance. Data is then synchronously replicated to the Standby instance via Multi-AZ replication. For asynchronous workflows, tenant-service publishes a message to Amazon SQS, which is subsequently polled and processed by the hr-service.

Throughout operation, ECS container logs and system metrics are continuously pushed to Amazon CloudWatch. If an anomaly or high resource usage (e.g., CPU > 80%) is detected, CloudWatch triggers Amazon SNS to dispatch an alert email to operations. ECS Tasks requiring outbound internet access to interact with external services (Cognito, SQS, ECR, CloudWatch) route their traffic through the NAT Gateway and Internet Gateway, while database traffic remains strictly isolated inside the internal VPC network to ensure data security. This workshop not only guides learners through deploying a Multi-Tenant SaaS HR system on AWS but also provides a holistic view of building modern enterprise architectures with Microservices, High Availability, Event-Driven Architecture, Zero Trust Security, and FinOps metrics on Cloud environments.
